# DCjanus Writing Style

用于生成或改写中文技术文本时，尽量贴近 DCjanus 的个人写作风格。这个 skill 只管文风和表达取舍；事实、代码、平台操作仍以当前任务上下文和其它对应 skill 为准。

## 总体气质

- 基本形象是克制、专业、可信的工程师。文字应像一个经历过真实系统复杂性的工程师在记录问题，而不是在做营销、布道或情绪输出。
- 可以有人味，允许出现“我之前遇到过”“这件事给我的印象比较深”这类经历感，但不要把文本写成朋友圈、鸡汤、段子或过度洒脱的随笔。
- 叙述要冷静、准确，优先解释现场约束、参与方、链路、行为后果和取舍原因。不要为了制造戏剧性夸大事故、风险或结论。
- 默认中文；保留必要英文技术名词、协议名、Header、API、命令、配置项和代码标识，不强行翻译。

## 结构习惯

- 先给读者必要背景，再进入具体故事或技术细节。背景不宜过长，但要让不在现场的人知道关键参与方、系统边界和问题为什么成立。
- 严肃叙述、复杂经历复盘或技术背景较重时，可以在段落或小节开头先用一句轻量概述说明本段大意，让扫读用户先建立预期，再进入细节。不要每段机械加摘要，避免显得重复或像报告模板。
- 讲经历时先找读者理解问题所需的最小路径，不要机械套固定结构。可以从背景、异常现象、常见误区或最终判断切入，只要后续能自然补齐正常假设、触发条件、问题后果和取舍理由。
- 对复杂业务背景，先拆参与方和整体架构，再展开事件。避免一上来塞进多个系统名、Header、网关、代理和权限规则。
- 不要突然抛出公司内部平台、系统名或组织抽象。先说明它解决什么问题、为什么在这篇文章里需要出现，再给名字或简短定义。读者不需要知道完整平台架构，但需要知道这个概念和当前论点的关系。
- 结尾应收束到工程判断或经验，而不是口号。可以指出更稳妥的方向，但不要写成万能清单或绝对定律。

## 可吸收的科技论文写法

- 把“逻辑路径”排在“经历发生顺序”前面。论文写作会先找最重要结论，再组织证据顺序；写工程经历时也应优先让读者顺着问题理解，而不是复刻排查时间线。
- 小节和段落开头尽量承担 topic sentence 的作用：先说明本段主张或作用，再给证据、例子和细节。扫读时只看小节标题、段首句和结尾，也应大致知道文章主线。
- 每段只推进一个主要想法。一个段落如果同时解释背景、配置、事故、修复和反思，读者需要自己拆结构；应拆成多个段落或小节。
- 使用 old-to-new 的信息顺序：句首承接读者已经知道的对象，句末放新的判断、例外或风险。这样读起来像沿着已有上下文前进，而不是不断被新名词打断。
- 重要判断放在容易被看见的位置。段首适合建立预期，段尾适合放结论、风险或转折；不要把核心结论塞在长句中间。
- 让主语和谓语靠近。避免在主语和动作之间插入太长的限定、背景和括号说明；必要时拆句，把补充信息放到下一句。
- 优先使用能表达动作的动词，少把动作名词化。比如优先写“平台覆盖 Header”“服务误判来源”，少写“Header 覆盖行为的发生”“来源判断的错误形成”。
- 主动语态和被动语态按信息焦点选择。需要强调责任、动作来源时用主动；动作执行者不重要、未知或容易干扰主线时，可以用被动。
- 不照搬论文格式，也不追求论文式完备。核心概念和关键因果不能错漏；周边概念可以一笔带过。默认读者有检索能力，不需要在文中替他们补完整背景百科。

## 句子与节奏

- 避免连续使用过长的长难句。必要时省略主语，或把因果链拆成两三句。
- 也不要过度使用短句。短句只用于强调关键判断；整篇都用短句会显得像营销号或演讲稿。
- 少用“首先、其次、最后”式模板化连接，除非确实是在列举步骤。更自然的推进可以使用“问题在于”“但它没有解决另一个现实问题”“再看另一个场景”。
- 少用“不是……而是……”“并不是……而是……”“不在于……而在于……”这类强对照句式。它们偶尔有用，但连续出现会有明显 AI 味；能改成直接陈述时，优先直接说结论。
- 控制重复。已经解释过的概念，后文只在需要推进判断时简短回扣，不要换说法反复铺陈。
- 形容词要少而准。少用“非常、极其、巨大、致命、灾难性”等强化词；确有严重影响时，用具体行为和后果说明。

## 技术表达

- 先讲可见行为和业务后果，再讲实现术语。读者应该先知道“发生了什么”，再知道“为什么这样发生”。
- 对 Header、协议、网关、代理、缓存、权限等主题，区分“语义”“实现约定”“安全边界”和“业务判断”。不要把这些概念混成一个词。
- 需要给例子时，用最小但真实可信的示例。IP、域名、Header 值可以匿名化，但链路关系要清楚。
- 能用现成代码、配置片段、命令输出或真实数据形态建立直觉时，优先展示这些具体材料。概念解释够读者继续理解即可，不为了完整性深入展开陌生概念，避免打散注意力。
- 严谨性可以向阅读流畅性妥协，但只限周边细节。核心概念、关键边界、因果关系和风险结论必须准确；可省略的是旁支背景、完整分类、历史来源和读者能自行检索的资料。
- 对安全、权限、线上系统问题，默认表达成风险边界和工程取舍，不做绝对化保证。比如用“更稳妥”“更保守”“可以缓解”，少用“一定安全”“彻底解决”。
- 如果一个规则会带来副作用，也要写出来。比如更保守的安全判断可能暴露配置错误、让部分内部调用需要修配置或补鉴权。

## 应避免的风格

- 不要写成营销号标题和段落：过多反问、悬念、金句、排比、夸张转折都不合适。
- 不要写成安全培训 PPT：大量 checklist、口号式“必须/千万不要/最佳实践”会削弱经历记录感。
- 不要写成纯规范文档：如果用户要的是文章或分享，应保留现场、时间顺序和人的观察。
- 不要过度口语化：可以自然，但不要使用大量网络语、调侃、自嘲、emoji 或随意感叹。
- 不要把过程中的探索、失败尝试、临时想法混进最终文章，除非它们本身是故事或判断的一部分。

## 改写流程

1. 先抽取原始材料中的业务背景、关键参与方、链路、异常输入、误判条件和最终教训。
2. 再决定文章重心：经历记录、技术解释、复盘、对外说明或短分享。不同重心会影响细节密度。
3. 写初稿时先保证结构和事实清楚，不急着追求文采。
4. 复读一遍，删掉重复主语、过度解释和模板句；拆开难读长句；保留少量有经历感的句子。
5. 最后检查语气：应当专业、克制、有人味，但不洒脱、不煽动、不端着。

## 快速校验

交付前用这几个问题自查：

- 读者是否能看懂关键参与方和链路？
- 是否先讲清楚了问题发生的条件，而不是直接给结论？
- 有没有把一个复杂系统问题写成简单道德判断或口号？
- 有没有连续长句、重复解释或营销号式短句？
- 有没有过度使用“不是……而是……”这类模板化强对照？
- 有没有保留适度经历感，而不是变成纯规范或纯教程？
