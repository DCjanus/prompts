# Change requests

适用于 Issue、PR/MR 与 inline review reply。先检查平台模板、仓库贡献规范、必填字段、标签、目标分支、squash 策略和当前资源状态；仓库规则优先。

## 语言与共同原则

- 用户明确指定语言时优先遵循；否则跟随仓库模板、贡献文档和 maintainer 的主要交流语言。没有可靠信号时使用英文。
- 先稳定语义、技术立场和承诺强度，再润色或翻译。
- 只描述当前需要提交给 reviewer/maintainer 的最终信息，不暴露本机绝对路径、home 目录、agent 工作区或临时正文文件。
- 更新已有标题或正文前先读取当前内容；根据最终状态重写完整文案，不在过时正文上机械追加补丁。

## 平台引用

- GitHub reviewer-facing 文案引用同仓库 commit 时，写裸的完整 40 位 SHA，由 GitHub 自动链接和折叠；不要包反引号或手写 Markdown URL。
- GitHub 跨仓库 commit 使用 `owner/repo@完整 SHA`。
- GitLab reviewer-facing 文案中的关联资源使用完整 Markdown URL，兼容自建实例和跨项目引用。

## Issue

- 独立 Issue 不要求本地工作区干净、分支已推送或存在 base/head diff；只有 Issue 明确基于本地改动时才检查相关代码、分支和 diff。
- 有模板时严格遵循模板，不跳过必填字段、标签、版本、复现信息或检查项。
- 没有模板时按目标选择结构：Bug 聚焦实际行为、预期行为、复现条件和影响；Feature 聚焦问题、目标和必要约束；其它 Issue 按目的组织。
- 不把 PR/MR 的固定结构机械套到 Issue，也不要为了形式完整添加空章节。

## PR/MR

- 先确认 base/head 或 target/source，`git fetch` 后以最终净变化为依据检查：

```bash
git diff --stat <base-or-target>...HEAD
git diff --name-status <base-or-target>...HEAD
```

- 正文只描述最终净变化，不记录中间提交顺序、调试过程、失败尝试、临时方案、merge/rebase 过程或最终 diff 已不存在的行为。
- 默认正文只包含：
  - `## Why`：1-2 条短句说明问题或目标。
  - `## What`：1-3 条说明主要行为变化，不罗列琐碎实现细节。
- 只有最终净变化确实破坏既有用法时，才在语义化标题中使用 `!`，并增加独立的 `## BREAKING CHANGE` 章节，分别说明影响范围与迁移方式。
- 仅在确有额外信息时添加 `## Risks` 或 `## Notes`。

## Validation Gate

默认不写 `## Validation`；不确定时也省略。只有同时满足以下条件时才添加：

1. 证据不在 final diff、平台 CI/checks、pipeline 或标准合入门槛中自然可见。
2. 证据会改变 reviewer 对风险的判断，而不是证明执行过测试。
3. 证据描述最终交付行为，而不是探索过程或临时调试。
4. 证据可以用行为结果表达，而不是命令流水账。

普通 lint、fmt、typecheck、build、测试、coverage、CI 状态、命令清单和测试数量都不要写入 Validation。允许的证据包括 CI 无法覆盖的真实环境行为、迁移 dry-run、性能数据、兼容性矩阵、安全边界或刻意构造的 red/green 回归结果。只写验证的行为和结果。

## Inline review reply

目标是用最低阅读成本消除 reviewer 对处理结果、设计理由或剩余风险的不确定性。

1. 先确定实际意图：接受并落实、部分落实、维持现状、推迟处理，或仅补充一种可能性。开头直接表达处理结果，不混淆决定、判断和开放想法。
2. 把原始评论、所在 diff 与当前 thread 视为共享上下文，不复述 reviewer 已经看得到的信息。只补充验证当前决定所必需、但无法从上下文直接得出的改动、约束、理由或风险。
3. 让解释深度与验证成本匹配。改动本身足以回答意见时，只给结果和可定位引用；涉及设计判断时，说明决定性约束、所选方案、实质性代价及其可接受依据。替代方案只有在有助于理解决定时才提及。
4. 区分事实、推断、判断与承诺。使用与决策相关的规模、频率、边界条件或系统不变量支撑风险判断；不确定的想法不要写成实施计划，也不要在并不需要答复时写成提问。
5. 假定 maintainer 熟悉仓库，但不了解未公开的讨论过程。省略仓库常识，解释隐藏或跨文件的因果关系，确保所有指代和结论都能从公开 thread 还原。
6. 使用简洁、专业且与仓库一致的语言。每个段落只承担一个沟通目的，并按平台渲染调整结构。
7. 发送前重读完整 thread，确认回复针对原始意见、引用最终代码，且没有把旧草稿或中间方案当成当前结论。
8. 仅修正文案时编辑原回复；出现新的独立信息时再追加回复。未经用户明确授权不要发送或 resolve。
