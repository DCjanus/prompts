# ADR 格式

ADR 放在 `docs/adr/` 下，使用连续编号：`0001-slug.md`、`0002-slug.md`，以此类推。

仅在需要第一条 ADR 时创建 `docs/adr/`。

## 模板

```md
# {简短的决策标题}

{用 1–3 句话说明背景、最终决定及其原因。}
```

仅此而已。ADR 可以只有一个段落；价值在于记录做出了什么决定以及为什么，而不是填满固定章节。

## 可选章节

仅在确实有价值时加入；大多数 ADR 都不需要：

- **Status** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）：适合可能被重新审视的决策。
- **Considered Options**：仅在被否决的替代方案值得保留时使用。
- **Consequences**：仅在需要记录不明显的后续影响时使用。

## 编号

扫描 `docs/adr/` 中已有的最大编号，然后递增一位。

## 何时建议创建 ADR

必须同时满足：

1. **难以逆转**：日后改变决定的成本显著。
2. **缺少背景便难以理解**：未来读者会疑惑为何这样设计。
3. **源于真实取舍**：确实存在替代方案，并基于明确理由做出选择。

容易改变、显而易见或不存在真实替代方案的决定都不需要 ADR。

### 符合条件的例子

- **架构形态。** 使用 monorepo；写模型采用事件溯源，读模型投影到 Postgres。
- **上下文之间的集成模式。** Ordering 与 Billing 通过领域事件通信，而非同步 HTTP。
- **带来锁定成本的技术选择。** 数据库、消息总线、认证提供商、部署目标；不包括每一个普通依赖库。
- **边界与范围决策。** Customer 数据由 Customer 上下文拥有，其它上下文只通过 ID 引用。
- **刻意偏离常见方案。** 因为明确原因使用手写 SQL 而不是 ORM，避免后来维护者误以为需要“修正”。
- **代码中不可见的约束。** 因合规要求不能使用 AWS；因合作方协议要求响应时间低于 200ms。
- **不明显的否决理由。** 比较 GraphQL 与 REST 后基于细微但重要的原因选择 REST，避免数月后重复讨论。
